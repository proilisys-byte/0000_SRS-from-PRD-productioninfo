#### [기존 필수 지침 유지]
- **목적:** 설계 스펙(API, DB)과 구현 코드 간의 불일치로 인한 런타임 500 에러를 배포 전 CI/CD 단계에서 원천 차단한다.
- **적용 범위:** Prisma Schema(`schema.prisma`), 에러 코드 정의서(`src/constants/error-codes.ts`), Zod 검증 스키마
- **필수 규칙:**
  - 기획 스펙 문서의 DB Nullable 여부와 Prisma Schema의 `?` 기호가 100% 매칭되어야 한다.
  - 서버에서 반환하는 에러 응답은 반드시 `error-codes.ts`에 선언된 Enum 값만 사용해야 하며, 하드코딩된 문자열을 응답해서는 안 된다.
  - PR 생성 전 로컬에서 `npx prisma validate`를 반드시 실행한다.
- **금지 사항:** 프론트/백엔드 Zod 타입을 별도로 선언하여 관리하는 행위 금지 (`src/lib/validation/` 내 단일 파일에서 Export 한 것을 양쪽이 Import 할 것).
- **예외 처리:** DB 스키마 긴급 수정이 필요한 경우, 코드가 아닌 기획 스펙 문서(v5)를 선행 수정한 후(Doc-first), PR에 반영한다.
- **완료 기준:** CI 단계의 `Type Check`와 `Prisma Validate` 스텝이 통과하지 않으면 PR Merge 버튼이 비활성화됨.
- **테스트 기준:** Prisma 파일 일부를 훼손하고 `npx prisma validate` 실행 시 에러가 나며 터미널 스크립트가 종료(Exit 1)되는지 확인.
- **초보자가 자주 틀리는 포인트:** DB에는 String으로 만들고, Zod Validation에서는 Number로 짜서 프론트엔드 연동 테스트 당일에 대규모 타입 충돌이 일어남.
