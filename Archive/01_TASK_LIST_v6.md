# PRO ILI SMART - 개발 태스크 목록 v6 (실행 가능한 개발 Task 명세서)

## 1. 총평
기존 v5 보완 문서의 기술적 지침과 운영 규칙은 훌륭하나, 실제 개발자가 '무엇을 보고, 누구에게 승인받으며, 언제까지, 무엇을 산출해야 하는지'에 대한 관리적 실행 지침이 누락되어 있다. 따라서 기존 가이드라인에 1) 입력자료/선행조건, 2) 산출물 정의, 3) 일정/우선순위, 4) 담당자/책임/승인 체계라는 4대 행동 지표를 강제 주입하여, 문서만으로도 개발 라이프사이클 전체가 통제되도록 완벽한 '실행 가능한 개발 Task 명세서'로 승격시킨다.

## 2. 보완 원칙
- **실행 가능성 극대화:** 모든 Task는 즉각 실행 가능한 수준의 명세로 구성한다.
- **물리적 차단 및 증빙 강제:** 스크린샷, 코드 링크 등 물리적 증빙을 통해서만 Task 완료를 인정한다.
- **책임 소재 명확화:** 누가 작업하고, 누가 리뷰하며, 책임자와 운영 담당자가 누구인지 단정적으로 명시한다.
- **기존 룰셋 유지:** v5 보완안에 작성된 필수 규칙, 금지 사항, 테스트 기준 등은 100% 원형을 보존하되, 명령형/단정형으로 톤앤매너를 일치시킨다.

#### [기존 필수 지침 유지]
- **목적:** 로컬 테스트 데이터가 운영 DB를 오염시키는 사고를 원천 차단하고, Vercel 환경에서의 무중단 배포 기준을 세운다.
- **적용 범위:** `.env` 파일 분리, Vercel 대시보드 환경변수 세팅, CI/CD 배포 파이프라인
- **필수 규칙:**
  - 개발 환경은 `.env.local` 파일만 사용하며, `DATABASE_URL`은 무조건 로컬 또는 Dev용 DB 주소만 기입한다.
  - 운영 DB 주소 및 Production Auth Key는 코드베이스에 절대 두지 않고 **Vercel 대시보드 Environment Variables** 메뉴에만 직접 입력한다.
  - 배포 파이프라인 순서: `1) tsc 타입 검사 → 2) ESLint 검사 → 3) Vercel Build → 4) npx prisma migrate deploy` 순서를 CI 상에 강제한다.
- **금지 사항:**
  - 로컬에서 `npx prisma db push`를 운영 DB에 연결해 쏘는 행위 **(발견 즉시 권한 압수)**.
  - `NEXT_PUBLIC_` 접두사가 붙은 환경변수에 비밀키를 할당하는 행위 절대 금지.
- **예외 처리:** 배포 스크립트 실행 중 Prisma 마이그레이션이 실패할 경우, Vercel 배포를 즉각 Cancel하고 이전 커밋으로 Revert 한다.
- **완료 기준:** GitHub 메인 브랜치 Merge 시 Vercel 빌드 로그에 타입/린트 에러가 0건이어야 배포가 승인됨.
- **테스트 기준:** 로컬 서버 구동 시 `process.env.NODE_ENV`가 `development`로, 운영 URL 접속 시 `production`으로 찍히는지 확인.
- **초보자가 자주 틀리는 포인트:** 운영 Vercel 환경변수에 `DATABASE_URL`을 잘못 넣고 `npx prisma generate`를 로컬에서 실행한 폴더를 통째로 올려 클라이언트 에러를 유발함.

#### [기존 필수 지침 유지]
- **목적:** 토큰 탈취 및 권한 우회를 시스템 단에서 방어하기 위해 미들웨어에서의 엄격한 검증 패턴을 강제한다.
- **적용 범위:** `src/middleware.ts` 및 API 권한 래퍼 함수(`src/lib/auth/withAuth.ts`)
- **필수 규칙:**
  - 미들웨어는 최상단에 위치하며, `matcher: ['/((?!api/public|_next/static|_next/image|favicon.ico).*)']` 정규식을 명시하여 정적 파일 검증을 스킵한다.
  - HTTP Request 헤더의 `Authorization: Bearer {토큰}` 존재 여부와 유효성을 반드시 JWT 검증 라이브러리(또는 Supabase 헬퍼)로 Decode 한다.
  - 디코드된 페이로드의 `role` 값과 API 스펙의 필요 권한을 비교하여 접근을 제어한다.
  - Mock Auth 모드 해제 시, `.env`의 `NEXT_PUBLIC_AUTH_MODE=real`로 변경하고 클라이언트 부트스트랩 시 `localStorage.clear()`를 강제 실행한다.
- **금지 사항:** 미들웨어 없이 각 API 라우트 내부에서 파편화된 방식으로 토큰을 확인하는 행위 절대 금지 (단일 진실 원칙 위배).
- **예외 처리:** 토큰이 만료되었으나 Refresh Token이 유효한 경우 재발급 후 응답 헤더에 새 토큰을 내려주고 비즈니스 로직을 속행한다.
- **완료 기준:** 유효하지 않은 토큰 혹은 권한 부족 시 로직 실행 전 즉시 `HTTP 401(Unauthorized)` 또는 `HTTP 403(Forbidden)` 응답 반환.
- **테스트 기준:** Postman에서 헤더 없이, 혹은 임의 조작된 토큰으로 `/api/v1/users` 호출 시 정확히 401 에러를 응답하는지 검증.
- **초보자가 자주 틀리는 포인트:** 미들웨어에서 DB를 직접 조회(Prisma Client 사용)하려다 Edge Runtime 제약에 걸려 서버리스 크래시를 유발함. (토큰 페이로드만으로 검증할 것).

#### [기존 필수 지침 유지]
- **목적:** 운영 장애(서버 다운, 500 에러 폭주) 발생 시 초보자가 상급자 승인 대기 없이 기계적으로 즉시 복구할 수 있는 권한과 절차를 부여한다.
- **적용 범위:** Vercel 배포된 프로덕션 서버, Supabase 프로덕션 DB 전체
- **필수 규칙:**
  - **장애 인지 시 조치 원칙:** 발견자(초보 개발자 포함)는 원인 분석을 생략하고 **Vercel 대시보드에서 마지막 정상 배포본으로 즉각 [Instant Rollback] 버튼을 누른다.**
  - **사후 전파:** 롤백 직후 사내 슬랙 `#ops-alert` 채널에 `@channel [긴급 롤백 완료] 일시 / 발생 증상 / 롤백 타겟 커밋`을 양식에 맞춰 즉시 보고한다.
  - DB 데이터 손실이 동반된 경우, 롤백 후 Supabase 대시보드의 PITR(Point-in-Time Recovery) 기능을 이용해 배포 10분 전 시점으로 복원한다.
- **금지 사항:** 장애가 터진 라이브 서버를 대상으로 로컬에서 수정한 코드를 `Hotfix`라는 이름으로 즉흥 배포하는 행위 **절대 금지**.
- **예외 처리:** 써드파티 API(예: 결제 PG, AI 모듈) 글로벌 장애 시 서버 롤백 대신 `NEXT_PUBLIC_MAINTENANCE_MODE=true`를 활성화해 점검 페이지를 띄운다.
- **완료 기준:** 장애 발생 후 15분 이내에 이전 정상 상태로 서비스가 재개되어야 함.
- **테스트 기준:** Staging 환경에서 치명적 에러 코드를 고의 배포한 후, 5분 내에 롤백 버튼으로 원상 복구하는 모의 훈련(Dry Run) 통과.
- **초보자가 자주 틀리는 포인트:** 서버가 뻗었는데 혼자 로그 보면서 고쳐보려다 복구 골든타임(15분)을 날려먹음. (장애는 고치는 게 아니라 먼저 '되돌리는' 것임).

  - 단일 요청 Payload 크기를 최대 5MB로 제한하며, 라우트 앞단 설정(`body-parser` limit 등)에서 즉각 413 에러로 잘라낸다.
  - 파일 파싱 시 전체 내용을 한 번에 배열 변수에 담지 말고, 파일 Stream API 기반의 Chunk 단위 처리를 강제한다.
  - 최대 허용 행 수(1,000행) 설정 및 파싱 에러 발생 시 정상 행은 Commit하고 에러 행 번호만 모아 "부분 성공"으로 응답한다.
- **금지 사항:** 10만 줄짜리 CSV를 통째로 읽어 `Promise.all` 로 한 번에 DB Insert를 날리는 행위 절대 금지.
- **예외 처리:** 부분 성공/실패 시 프론트엔드에 `{ success_count: 990, error_rows: [5, 12, 105], message: "일부 행 처리 실패" }` 규격으로 반환.
- **완료 기준:** 1,001줄짜리 CSV 파일 전송 시 스트림 버퍼 단계에서 HTTP 413 (Payload Too Large)를 뱉고 튕겨냄.
- **테스트 기준:** 빈 행이 섞인 5MB 파일 전송 시 정상 행만 DB에 적재되고, 서버 램(RAM) 사용률 폭증이 없는지 모니터링 확인.
- **초보자가 자주 틀리는 포인트:** 서버 응답 타임아웃(Vercel 기본 10초)을 간과하여, 파싱은 성공했으나 프론트엔드에 504 Gateway Timeout 에러를 반환해 유저가 다시 업로드하게 만듦.
