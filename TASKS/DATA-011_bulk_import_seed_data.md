# [DATA-011] Bulk Import 기준 데이터(Seed Data) 명세

## 1. 문서 목적

*   **문서 필요성:** Sprint 1 핵심 운영 기능인 Bulk Import를 성공적으로 데모하고 QA 및 모니터링하기 위해, 시스템 초기 구동 시 반드시 적재되어야 하는 공통 기준 데이터(Seed Data)의 구조와 시나리오를 정의한다.
*   **Sprint 1 중요성:** 백엔드(API), 프론트엔드(UI), 테스트(QA) 파트가 각자 다른 가짜 데이터를 머릿속에 그리고 있으면 통합 시 반드시 충돌이 발생한다. 하나의 단일 데이터 세계관(Single Source of Truth)을 구성하여 병렬 개발과 검증의 일관성을 확보한다.
*   **후속 구현 문서와의 관계:** 본 문서는 이후 실제 DB 초기화 스크립트(Prisma seed 등)를 작성할 개발자와, E2E 테스트 스크립트를 짤 QA, 그리고 Mock 서버를 구축할 프론트엔드에게 공통 데이터 딕셔너리 역할을 한다.

## 2. Seed Data 설계 원칙

Seed 데이터는 단순한 빈칸 채우기가 아니라, 실제 운영 환경에서 일어날 수 있는 권한과 상태 전이의 리허설 세트여야 한다.

### 표 1. Seed Data 설계 원칙
| 원칙 | 설명 | 적용 이유 | 관련 문서 | Sprint 우선순위 |
| :--- | :--- | :--- | :--- | :--- |
| 현실성 확보 | "test", "aaa" 같은 데이터 지양, 실제 도메인에 가까운 "용접 공정 검사", "Q2 품질 리포트" 등 형태 유지 | 시연(Demo) 및 UI/UX 검증 시 실제 운영 느낌 제공 | `00_PRD_v1` | P1 (Sprint 1) |
| 권한 및 경계 검증 보장 | 최소 2개 이상의 테넌트(Tenant)와 역할(Role)별 계정을 명확히 구분하여 적재 | RBAC 및 데이터 격리(Isolation) 결함 탐지 필수 | `COM-AUTH_v1` | P1 (Sprint 1) |
| 시나리오 재현성 | 성공뿐 아니라 부분 성공, 실패, 재처리 등 다양한 생애주기(Lifecycle) 데이터 포진 | 예외/오류 UI 테스트 및 모니터링 경고 테스트용 | `TEST-ADM-001` | P1 (Sprint 1) |
| 개인정보 철저 배제 | 이메일은 `@example.com` 사용, 전화번호는 `010-0000-0000` 형태 고정 등 실제 정보 사용 절대 금지 | 운영/QA 시 외부 유출 및 보안 위반 리스크 원천 차단 | `DATA-SCHEMA_v1` | P1 (Sprint 1) |

## 8. Seed 데이터셋 구성 방식

단일 테이블이 아닌 RDBMS(또는 연결된 객체) 관점의 구조적 연계를 정의한다.

### 표 7. Seed 데이터셋 구성 방식
| 데이터 레벨 | 설명 | 포함 항목 | 활용 목적 | 비고 |
| :--- | :--- | :--- | :--- | :--- |
| 마스터 (Header) 데이터 | Job 자체의 메타정보 (리스트 화면용) | `id`, `filename`, `status`, `total_count`, `tenant_id` | 목록 렌더링, 권한 검사 매핑 | |
| 에러 (Row) 데이터 | 부분 성공/실패 시 매핑되는 오류 내역 컬렉션 | `job_id`, `row_number`, `error_type`, `raw_data` | 부분 성공 시 에러 목록 드릴다운(표 렌더링) | |
| 에러 (Field) 데이터 | 특정 행(Row) 내 어느 열(Column)이 틀렸는지 상세 | `row_id`, `field_name`, `error_message` | "수량 필드는 숫자여야 합니다" 등 상세 피드백 노출 | |
| 요약 (Summary) 데이터 | 대시보드 상단 집계를 위한 미리 계산된 통계 | `success_count`, `failed_count`, `processing_time` | 상세 진입 전 퀵 뷰(Quick View) 데이터 | |
