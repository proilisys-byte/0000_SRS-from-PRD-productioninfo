# 프로젝트 컨텍스트 (Project Context) v1.0

## 1. 문서 목적
이 문서는 PRO ILI SMART 프로젝트의 기획(PRD), 기술 명세(SRS), 그리고 합의된 의사결정 로그(Decision Log)를 통합한 **최상위 기준 문서(Single Source of Truth)**이다. 
본 문서는 "우리가 무엇을, 왜, 어떤 제약 속에서 만드는가"를 명확히 정의하며, 하위 모든 기획 및 엔지니어링 문서(TASK, API, DB 등)는 본 문서의 원칙과 범위를 절대적으로 위반할 수 없다. 
경영진부터 개발팀, 운영/QA 조직에 이르기까지 프로젝트의 방향성을 동기화하는 기준선 역할을 수행한다.

## 3. 프로젝트 한 줄 정의
> "현장의 수기 데이터를 Zero-UI로 디지털화하고, AI 기반 문서 매핑을 통해 원청 심사 및 부적합(NC) 대응의 비용(COPQ)을 획기적으로 감축하는 초저비용 SaaS 플랫폼"

## 4. 문제 정의
제조 중소기업은 글로벌 원청의 엄격한 품질/ISO Audit에 대응하기 위해 막대한 인력과 야근(히든 팩토리 비용)을 소모하고 있다.
현장 작업자는 오염/소음 환경에서 수기나 복잡한 태블릿 입력으로 인해 데이터 기록을 회피하거나 누락하며, 이로 인해 신뢰할 수 없는 로우 데이터가 생성된다. 
기존 MES/ERP 솔루션은 도입 비용이 비싸며, 현장 작업자의 입력 마찰(Friction)을 해결하지 못하는 근본적 한계를 지닌다.

## 5. 해결하려는 핵심 가치
1. **Zero-UI 혁신**: 음성과 비전 AI를 통해 키보드 입력 없이 소음·장갑 환경에서도 마찰 없는 데이터 입력 실현.
2. **Audit 대응 시간 90% 이상 단축**: 수기 데이터를 모아 엑셀로 짜맞추는 120시간의 야근을, AI 자동 매핑을 통해 10분 내 PDF 생성으로 단축.
3. **긴급 NC 시정 자동화**: 원청의 부적합 통보 시 즉각적인 시정 조치 초안 생성 및 무결성 기반 진행 추적.
4. **COPQ 가시화**: 숨겨진 품질 실패 비용(재작업, 과잉생산 등)을 금전적 가치로 환산하여 ROI 기반 의사결정 지원.
5. **저비용 SaaS 전환**: 고정 인프라 유지비를 제거한 Serverless 아키텍처로, 영세 기업도 즉시 도입 가능한 $0 초기 인프라 달성.

## 12. 상위 의사결정 반영 사항

* `08_DECISION_LOG_v1.md`에 의거, 기능적 이상향보다 기술적 근간(기반 공사)을 0순위로 둔다.
* 초기 데이터 적재 불능 상태를 막기 위해 Bulk Import를 Sprint 1 코어로 격상하여 시연을 방어한다.
* AI 연동은 단계적(STT → Vision → NC → Lean)으로 전개하여 아키텍처 부하 및 개발 지연을 회피한다.
* 단순한 기능 완성을 넘어, 컴플라이언스(동의/노조합의) 미달성 시 배포를 중단하는 락업 기조를 따른다.

## 13. 문서 체계 운영 원칙

1. **Top-Down 통제**: 본 문서(`PROJECT_CONTEXT`)에 명시된 기능 범위와 제약사항을 하위 문서는 임의로 변경할 수 없다. (예: 하위 API 설계서에서 갑자기 FastAPI를 추가하는 행위 금지)
2. **단일 진실 공급원 (SSOT)**: 비즈니스 목적은 기획팀에 묻지 말고 본 문서를 기준으로 판별한다. 
3. **업데이트 규정**: 기획 변경이 발생할 경우, 최상위 문서인 이 문서를 먼저 개정한 후 파생 문서들을 동기화한다.

* **선후행 관계의 명시**: "Auth 및 DB 세팅 ➔ Bulk Import 기능 완성 ➔ 기준 정보 CSV 업로드 ➔ STT Zero-UI 수집 테스트 ➔ Smart Audit 리포트 통합 생성" 으로 이어지는 크리티컬 패스(Critical Path)를 다이어그램상에 강제로 연결하라.
* **규제 락업 블록 표시**: PIPA 동의 및 노조 합의는 개발 태스크 밖의 일정이지만, "MVP 배포(Release)" 노드 앞의 거대한 블로커(Blocker) 노드로 반드시 시각화할 것.

### 3. 세부 문서(DATA, API, F1~F4) 분해 시 주의사항
* **DATA 문서 작성 시**: Prisma 확장을 활용한 SQLite-PostgreSQL 직렬화 대응 방안과 Audit Log 무결성 제약(Insert-only)을 스키마에 명시할 것.
* **API/COM 문서 작성 시**: REST API 명세 형식을 피하고, Next.js Server Actions 폼 처리와 Route Handlers의 응답(Zod를 통한 검증 등) 구조로만 작성할 것.
* **F1~F4 문서 작성 시**: 긴급하게 배치된 Sprint 2, 3 기능이 Sprint 1 아키텍처를 뒤엎지 않도록, 기능 확장에 대비한 의존성 인터페이스 설계를 염두에 두고 작성할 것.
