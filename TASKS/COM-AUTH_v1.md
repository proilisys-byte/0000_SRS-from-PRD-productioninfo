# 공통 인증/권한 설계 명세서 (COM-AUTH) v1.0

## 1. 문서 목적
본 문서는 PRO ILI SMART 프로젝트의 사용자 인증(Authentication) 및 권한 부여(Authorization) 아키텍처를 정의하는 공통 설계 문서이다. 멀티테넌시(Multi-tenancy) 환경에서 데이터 격리를 보장하고, 규제 준수(Compliance)를 위한 감사(Audit) 추적성을 확보하며, 향후 확장 가능한 RBAC(Role-Based Access Control) 구조의 베이스라인을 제공하는 것을 목적으로 한다. 이 문서는 향후 API 엔드포인트 보안, UI 라우트 보호, 관리자 콘솔 설계, 그리고 모든 보안 테스트 시나리오의 상위 기준점이 된다.

## 2. 인증/권한 설계 원칙

### 표 1. 인증/권한 설계 원칙
| 원칙 | 설명 | 적용 이유 |
| :--- | :--- | :--- |
| **인증보다 접근 통제 우선** | "누가 접속했는가"보다 "접속한 자가 현재 맥락(Tenant/Site)에서 무엇을 할 수 있는가"를 더 엄격히 검증한다. | 데이터 유출의 대부분은 로그인 실패가 아닌 내부자의 과도한 권한 오남용에서 발생하기 때문. |
| **Tenant 경계의 절대성** | 어떠한 경우에도 A 고객사(Tenant)의 사용자가 B 고객사의 데이터에 접근할 수 없다. | B2B 엔터프라이즈 SaaS의 가장 기본적이고 치명적인 신뢰/보안 요건. |
| **최소 권한 부여 (Least Privilege)** | MVP 단계의 권한은 Admin과 Standard User로 최대한 단순화하되, 기본 상태는 거부(Deny-by-default)로 설정한다. | 복잡한 권한 체계는 초기 개발 속도를 저해하나, 너무 느슨한 권한은 대형 보안 사고를 유발함. |
| **모든 관리 행위의 감사 가능성** | 권한 변경, Bulk Import, 리포트 강제 수정 등 주요 행위는 반드시 `AuditLog`와 연결된다. | 규제 기관 감사 시 "언제/누가/무엇을" 변경했는지 즉각 증명해야 함. |
| **규제 선행주의** | 인증된 사용자라도 PIPA 동의 및 필수 노조 합의 조건이 누락된 경우 핵심 기능 접근을 시스템 레벨에서 차단한다. | 개발이 완벽해도 법적 요건 미충족 시 현장 배포(Lock-up)가 불가능하므로 이를 통제 시스템에 내장. |

## 3. 인증 범위와 적용 대상
본 설계는 PRO ILI SMART 플랫폼에 접속하는 모든 내부/외부 사용자를 대상으로 한다. 모바일 환경(STT/Vision 캡처) 및 웹 대시보드(관리자/오딧 리뷰) 모두 동일한 인증 파이프라인(Supabase Auth 기반)을 통과해야 하며, 모든 API 요청에는 유효한 세션 토큰(JWT)이 포함되어야 한다.

## 4. 사용자 유형 정의

### 표 2. 사용자 유형 정의
| 사용자 유형 | 설명 | 주요 사용 기능 | 제한 사항 |
| :--- | :--- | :--- | :--- |

| **Super Admin**<br>*(System)* | 플랫폼 운영사(PRO ILI) 관리자 | 신규 Tenant 프로비저닝, 글로벌 구독 관리, 글로벌 감사 로그 조회 | 개별 Tenant의 민감 데이터(음성, 사진 등) 원본 직접 열람 제한 (동의 필요). |

### 표 5. 인증 흐름
| 단계 | 설명 | 관련 주체 | 성공 조건 | 실패 시 처리 |
| :--- | :--- | :--- | :--- | :--- |
| **1. 로그인 요청** | ID/PW 또는 SSO로 자격 증명 제출 | 사용자 | DB에 정보 존재 및 비밀번호 일치 | `401 Unauthorized` & 로그인 실패 로그 |

## 10. 감사 추적 연계 방식

1. **인증과 `AuditLog`의 결합**: 
   - 모든 수정/삭제성 API 요청은 미들웨어 단에서 Access Token의 `user_id`를 파싱하여, 데이터 처리 성공 직후 무조건 `AuditLog`에 `actor_id`로 기록한다.
2. **보안 이벤트 분리**: 
   - 단순 시스템 사용 로그와 달리, 연속된 로그인 실패, 미인가된 권한 접근 시도, MFA 우회 시도 등은 `SecurityEvent`로 분류하여 Admin 대시보드에 즉각 경고를 발생시킨다.

### 표 9. 보안 이벤트 / 감사 연계 항목
| 이벤트 | 기록 대상 | 중요도 | 후속 조치 |
| :--- | :--- | :--- | :--- |
| 로그인 5회 연속 실패 | 사용자 계정 ID, IP, 시간 | 중간 | 10분간 해당 IP 차단 및 사용자 알림 |
| 타 Tenant 데이터 접근 시도 | 요청 API, 소속 Tenant, 대상 Tenant | CRITICAL | 즉각 세션 만료 및 Super Admin에 알림 (해킹 시도 간주) |
| 권한 승격 완료 | 대상 User, 승격시킨 Admin, 변경 전/후 권한 | 높음 | Audit Log 영구 보존 |
| 동의 철회 | 사용자 ID, 파기된 동의 유형 | 높음 | 연관된 원시 데이터(음성 등) 즉각 Soft Delete |

## 11. 규제 및 동의 반영 방식

- **PIPA (개인정보보호법) 게이트웨이**: 
  클라이언트가 STT/Vision 관련 라우트로 진입할 때 서버 사이드(Next.js 미들웨어 등)에서 `ConsentRecord`의 상태를 조회한다. 유효한 동의가 없다면 API 요청은 `403 Forbidden`을 반환하며 클라이언트는 동의 서명 폼으로 강제 리다이렉트된다.
- **노조 사전 합의 정책**: 
  기술적 코드가 아닌 'Tenant 단위 상태(Status)' 플래그로 관리한다. 특정 Site의 합의 플래그가 `FALSE`라면, 해당 Site의 어떠한 일반 사용자도 입력 시스템을 사용할 수 없도록 전면 락업(Lock-up)을 건다.

---

## 12. 예외 / 실패 / 차단 시나리오

- **토큰 탈취 대비**: Refresh Token Re-use Detection 기법을 적용하여, 동일한 Refresh Token이 두 번 사용되면 해당 계정의 모든 세션을 무효화한다.
- **클라이언트 측 변조 방어**: UI 상에서 버튼을 강제로 노출시켜 누르더라도, Server Actions나 API Route 레벨에서 `tenant_id`와 `role`을 재검증하므로 실제 부정한 데이터 조작은 불가능하다.

## 13. API / UI / ADM / TEST 문서에 미치는 영향

- **API**: 모든 엔드포인트는 인증 토큰 유무, Tenant 매칭 여부, Role 부합 여부를 검증하는 통일된 미들웨어 스택을 통과해야 한다.
- **UI**: Role 및 동의 상태에 따라 메뉴 렌더링이 동적으로 분기되며, 접근 불가 메뉴는 화면에서 은닉된다.
- **ADM**: 시스템 관리자는 이 문서에 정의된 `SecurityEvent`와 `AuditLog`를 모니터링할 수 있는 콘솔을 설계해야 한다.
- **TEST**: 권한 테스트 시나리오는 반드시 "허용된 자의 성공"뿐만 아니라 "권한 없는 자의 접근 시도 차단" 시나리오(Negative Test)를 1:1 비율로 포함해야 한다.

## 15. 다음 단계 작성 가이드

인증/권한 구조 설계가 완료됨에 따라, 개발 조직은 아래와 같은 구체화 단계로 넘어가야 한다.

1. **우선 작성 문서 Top 10**
   - `DATA-SUPABASE_RLS_v1.md` (Tenant/Site RLS 쿼리 스펙)
   - `API-AUTH_MIDDLEWARE_v1.md` (Next.js 미들웨어 라우트 가드 스펙)
   - `COM-ERROR_HANDLING_v1.md` (401/403 에러 발생 시 공통 응답 구조)
   - `API-BULK_IMPORT_v1.md` (Admin 전용 API)
   - `UI-CONSENT_FLOW_v1.md` (사용자 최초 로그인 시나리오 화면)
   - `ADM-USER_MANAGEMENT_v1.md` (Admin 사용자 초대 및 역할 할당 화면)
   - `F1-STT_RECORDING_v1.md` (권한 기반 음성 수집 컴포넌트)
   - `TEST-AUTH_RBAC_v1.md` (Role별 접근 통제 E2E 테스트 시나리오)
   - `NFR-MFA_ENFORCEMENT_v1.md` (관리자 강제 MFA 흐름 및 복구 프로세스)
   - `ADM-AUDIT_DASHBOARD_v1.md` (감사 로그 조회 콘솔)

2. **`ADM-*` 문서로 분해 영역**
   - 사용자 관리 (초대, 권한 변경)
   - 보안 모니터링 (SecurityEvent, AuditLog 뷰어)

3. **`API-*` 문서로 분해 영역**
   - JWT 파싱 및 인가(Authorization) 미들웨어 로직.
   - PIPA 동의 상태 갱신/철회 API.

4. **`TEST-*` 문서로 직결될 영역**
   - Tenant 경계 침범 테스트 (Cross-tenant Data Access Attempt).
   - 권한 에스컬레이션(Privilege Escalation) 방어 테스트.

5. **Opus 4.6 (아키텍트) 추가 검토 포인트 3가지**
   - **Supabase Auth vs NextAuth**: Supabase RLS 연동의 이점과 NextAuth의 소셜/커스텀 제공자 유연성 사이의 트레이드오프 최종 결정.
   - **MFA (aal2) 세션 갱신 UX**: 작업 중간에 `aal1` -> `aal2`로 강제 승격되어야 할 때 클라이언트 단의 매끄러운 팝업 처리 및 원래 작업 복귀 흐름.
   - **Rate Limiting과 Auth의 결합**: Brute-force 공격 방지를 위한 로그인 및 고위험 API(예: 비밀번호 변경) 대상의 IP/User 단위 Rate Limiting 전략.
