# TEST-S1_ACCEPTANCE_v1: Sprint 1 합격 기준 및 데모 통과 명세

## 1. 문서 목적

* **문서 필요성**: PRO ILI SMART 프로젝트의 Sprint 1에서 개발된 핵심 코어 기능(Auth, RBAC, Bulk Import, STT Zero-UI, Smart Audit, Audit Log)이 단순히 코드 작성을 넘어 "실제 시연 및 다음 Sprint 확장"이 가능한 품질에 도달했는지 명확한 판정 기준을 제시한다.
* **Sprint 1에서의 중요성**: 기반 공사와 데이터 흐름의 뼈대가 완성되는 Sprint 1의 통과 여부를 결정짓는 게이트웨이 문서이다.
* **다른 문서와의 관계**: 본 문서는 기능/운영 명세서(`00_PRD_v1`, `F1-AUDIT_v1`, `ADM-BULK_v1` 등)의 요구사항을 검증 관점으로 번역한 최상위 테스트 기준서이며, 향후 작성될 세부 QA 테스트 명세(`TEST-F1-AUDIT_v1`, `TEST-BULK_IMPORT_v1` 등)의 상위 지침이 된다.

## 2. Sprint 1 Acceptance 전략

### 표 1. Sprint 1 Acceptance 개요

| 구분 | 설명 | Sprint 1 적용 여부 | 판정 기준 | 비고 |
|---|---|:---:|---|---|
| **Pass** (합격) | 결함 없이 필수 폐쇄 루프 정상 동작 | O | 주요 기능(P1) 결함 0건, 비기능 기준 100% 충족 | 데모 통과 및 다음 Sprint 이관 가능 |
| **Conditional Pass** (조건부) | 데모 시연 및 후속 작업 진행에는 무리가 없는 경미한 결함(Minor) 존재 | O | P1 결함 0건, P2 이하 결함 3건 미만 | 결함 티켓 발행 후 Sprint 2 기술 부채로 이관 |
| **Fail** (불합격) | 핵심 기능 동작 불가 또는 데이터 무결성 파괴 (즉시 조치 요망) | O | P1 결함 1건 이상, 또는 데모 시나리오 완주 불가 | 배포 불가, Sprint 1 기간 연장 또는 핫픽스 |

* **핵심 검증 원칙**: "UI에서 버튼이 눌린다"가 아니라, "입력 ➔ 검증 ➔ 저장 ➔ 리포트 ➔ 로그 적재 ➔ 오류 처리"로 이어지는 백엔드 데이터 흐름의 완결성을 우선 검증한다.

## 15. Gemini 3.1 Pro 자체 검토 체크포인트

본 문서가 완벽한 Acceptance Document로 동작하기 위해 다음 10가지를 검토 및 준수하여 작성되었습니다.

1. **Sprint 1 인스코프/아웃오브스코프 구분이 명확한지**: Vision AI, NC, COPQ 등을 S1 필수 범위에서 철저히 배제(표 2)하여 데모 리스크를 낮추었는가.
2. **Pass / Conditional Pass / Fail 기준이 판정 가능한지**: 막연한 "잘 동작함"이 아닌 결함 등급(표 9)과 수량 기반의 구체적 허용치를 제시했는가.
3. **Bulk ➔ STT ➔ Audit ➔ Report ➔ Log 흐름이 반영되었는지**: 단일 기능 검증을 넘어 4가지 핵심 단계가 파이프라인으로 연결되는 데모 시나리오(표 8)를 강제했는가.
4. **RBAC / Tenant / Site 경계 검증이 포함되었는지**: S1-TC-02 시나리오를 통해 타 테넌트 침범 차단 여부가 필수 통과 기준에 포함되었는가.
5. **공통 에러 스키마 검증이 빠지지 않았는지**: HTTP 응답 코드 및 `error.details` 객체 노출 여부를 합격 판정의 핵심 기준으로 잡았는가.
6. **비기능 기준이 과도하지도 부족하지도 않은지**: 10분 내 리포트 산출(KPI), 5초 내 STT 반응 등 Sprint 1에 타당한 수치(표 7)로 합격선을 지정했는가.
7. **데모 실패 조건이 명확한지**: "부분 실패 롤백 실패"나 "STT 수동 대체 불가" 등 앱 데모 자체를 멈추게 하는 치명적 조건을 정의했는가.
8. **증빙 자료 기준이 실제 제출 가능한 수준인지**: 동영상 녹화본, DB/API 로그 스냅샷 등 PM과 QA가 수집 가능한 구체적 산출물(표 10)을 지목했는가.
9. **Sprint 2 이후 기능이 과도하게 섞이지 않았는지**: 재처리 로직(Upsert) 고도화 등을 S1 실패 사유가 아닌 Conditional Pass의 후속 과제(표 11)로 밀어냈는가.
10. **QA / PM / 개발팀이 동일 기준으로 해석 가능한지**: 모든 문장이 단정적이고 행동 지향적이며, 모호한 형용사 배제로 부서 간 언어 충돌을 방지했는가.
