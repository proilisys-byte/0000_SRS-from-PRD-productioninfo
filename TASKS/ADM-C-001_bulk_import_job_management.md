# [ADM-C-001] Bulk Import Job Management Workflow

## 1. 문서 목적
본 문서는 **PRO ILI SMART** 프로젝트에서 관리자가 대량의 데이터를 업로드하고 처리하는 **Bulk Import Job**의 운영 처리(Management/Command) 흐름을 정의하는 명세서이다.

단순한 파일 업로드 기능이 아니라, 업로드 이후 검증, 저장, 실패 처리, 재시도, 취소에 이르는 전체 생명주기를 다룬다. 본 문서는 부분 성공을 정상적인 운영 시나리오로 편입시키고, 상태 전이의 기준을 제공하여 후속 조회(ADM-Q), UI, 테스트 문서가 공통된 운영 정책을 따르도록 가이드한다.

## 6. 핵심 업무 흐름

### 표 5. 핵심 업무 흐름
| 단계 | 설명 | 입력 | 처리 | 출력 | 실패 시 대응 | Audit Log 기록 여부 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **1. 업로드 요청 접수** | 관리자가 파일을 첨부하여 API 요청 | `RequestDto` (파일, 메타) | 파일 검증, Job 엔티티 생성, 초기 상태 저장 | Job ID, `PENDING` 상태 반환 | Http 400 에러 및 즉시 응답 | Y (Job Create) |
| **2. 파싱 및 사전 검증** | 워커가 파일을 파싱하고 헤더/컬럼 매핑 확인 | 파일 스트림 | 인코딩 확인, 헤더명 일치 여부 판별 | 파싱된 JSON 배열 | 상태를 `FAILED`로 변경, 원인 기록 | N |
| **3. 행 단위 검증** | 파싱된 데이터 행별로 비즈니스/타입 유효성 검사 | JSON 배열 | 빈 값, 포맷, 중복 여부 확인 | 성공 목록 / 실패 목록 분류 | 실패 행은 에러 목록에 적재 | N |
| **4. 저장 처리** | 검증을 통과한 데이터를 DB에 반영 | 성공 목록 JSON | Chunk 트랜잭션 수행 | DB Insert 완료 건수 | DB 레벨 예외 시 해당 Chunk 내 실패건 처리 | N |
| **5. 결과 집계** | 전체 처리 후 성공/실패 통계 업데이트 | 처리 결과 내역 | 전체/성공/실패 카운트 및 종료 시간 저장 | 상태를 `SUCCESS` 또는 `PARTIAL_SUCCESS`로 전이 | 통계 산출 실패 시 `FAILED` 전환 | Y (Job Complete) |
| **6. 실패 건 재처리 (S2)** | 관리자가 실패 건을 수정/선택하여 재시도 | `RetryRequestDto` | 재처리 대상 추출 후 3번 단계부터 재수행 | 신규 하위 Job 또는 상태 갱신 | 반복 실패 처리 | Y (Job Retry) |

## 7. Job 상태 정의

### 표 6. Job 상태 정의
| 상태 | 의미 | 진입 조건 | 종료 조건 | 운영자 액션 | 비고 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| `PENDING` | 시스템에 작업이 등록되어 큐 대기 중 | 업로드 API 호출 성공 | 파서가 작업을 가져감 | (취소 가능 - S2) | |
| `PROCESSING` | 파일 파싱, 유효성 검사, DB 반영 진행 중 | 워커가 작업 시작 | 모든 행 처리 완료 또는 예기치 않은 서버 크래시 | 진행률 모니터링 | 강제 취소 지양 |
| `SUCCESS` | 모든 데이터가 단 1건의 오류 없이 반영됨 | 전체 건수 == 성공 건수 | - | 이력 조회 | |
| `PARTIAL_SUCCESS` | 일부는 성공하고, 1건 이상의 실패 데이터가 존재함 | 처리 완료 및 실패 건수 > 0 | 재처리 완료 시 (S2) | 실패 원인 파악 및 다운로드 | **가장 주의 깊게 다룰 핵심 상태** |
| `FAILED` | 파일 포맷 오류, 0건 처리 등 배치 전체가 실패함 | 파싱 불가 또는 시스템 예외 | - | 파일 수정 후 재업로드 | |
| `CANCELED` (S2) | 관리자 요청으로 처리가 강제 중단됨 | 관리자가 취소 요청 | 취소 로직 반영 완료 | - | 데이터 일관성 주의 요망 |

| 처리 규칙/기능 | Sprint | 이유 | 선행조건 | 후속 영향 | 비고 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **Job 생성 및 PENDING/PROCESSING 상태 전이** | 1 | 핵심 워크플로우 구동 최소 조건 | DB/워커 설정 | 사용자 가시성 제공 | 필수 |
| **부분 성공 (PARTIAL_SUCCESS) 판정 및 통계** | 1 | "모 아니면 도" 처리의 한계를 극복하고 실무 적용 가능성 입증 | Chunk 처리 로직 | 에러 그리드 상세 조회 API | 가장 중요함 |
| **Tenant / Site 권한 기반 데이터 격리** | 1 | 멀티테넌트 사고 예방을 위한 최소한의 안전장치 | RBAC 통합 | Audit Log 연동 | 필수 |
| **실패 건 부분 재처리 및 강제 취소** | 2 | Sprint 1에서는 "오류 수정 후 재업로드"로 회피 가능하므로 우선순위 조정 | 하위 Job 관리 기능 | 복잡한 UI 액션 | 선택적 확장 |

## 14. 완료 기준(Definition of Done)
- **문서 완료 조건**: Job의 생성부터 처리 완료(성공/부분 성공/실패)까지 운영자가 판단하고 대응할 수 있는 상태 전이도와 액션 규칙이 누락 없이 문서화됨.
- **운영 흐름 정의 완료 조건**: '부분 성공' 시나리오에 대한 시스템 및 관리자의 대응 방식이 명확하며, 권한 오남용을 막기 위한 Audit 로깅 및 격리 기준이 정립됨.
- **후속 문서 전달 가능 조건**: 조회를 담당하는 `ADM-Q-001` 및 화면을 그리는 `UI-061` 개발자가 상태값을 어떻게 파싱하고 UI 분기(ex: "에러 내역 보기" 버튼 활성화)를 쳐야 할지 기준이 제시됨.
- **테스트 가능 조건**: `PARTIAL_SUCCESS` 도출을 위한 임의의 혼합 데이터(정상+오류) 업로드 테스트 시나리오를 QA가 작성할 수 있는 근거 제공.
