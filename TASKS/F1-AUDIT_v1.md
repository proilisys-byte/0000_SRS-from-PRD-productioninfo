# F1-AUDIT_v1: Smart Audit 코어 기능 명세

## 1. 문서 목적

* **필요성**: PRO ILI SMART의 "북극성 지표(10분 내 리포트 산출)"를 달성하기 위한 가장 핵심적인 비즈니스 기능인 현장 점검(Audit)의 디지털화 표준을 정의한다.
* **프로젝트 내 위치**: 본 기능은 시스템의 심장이자 가치 창출의 원천이다. STT Zero-UI로 현장 데이터를 수집하고, Bulk Import된 마스터 데이터를 기반으로 구조화하며, 최종적으로 Insert-only Audit Log와 엮여 무결성이 보장된 리포트를 산출한다.
* **기준 제공**: 후속 설계될 백엔드 API(`API-AUDIT_REPORT`), 프론트엔드 UI(`UI-AUDIT_WORKSPACE`), 그리고 테스트 명세(`TEST-F1-AUDIT`)의 상태 전이, 권한 검증, 에러 처리 기준을 강제하는 최상위 기능 스펙이다.

---

## 2. 기능 개요

* **한 줄 정의**: 현장 심사원의 음성(STT)과 표준 템플릿을 결합하여, 감사 시작부터 리포트 생성 및 승인까지의 전 과정을 자동화하고 추적하는 지능형 감사 코어 엔진.
* **기존의 문제**: 수기 점검표 작성, 엑셀 재입력, 수동 취합으로 인한 데이터 유실, 결과 왜곡, 리포트 작성 시간의 심각한 지연(평균 수일 소요).
* **기대 효과**: 현장에서 즉시 데이터 입력, 실시간 리포트 산출(10분 이내), 데이터 조작 원천 차단(감사 추적성 확보).
* **Sprint 1 최소 가치**: "표준 템플릿 기반의 점검 폼 로드 → STT를 활용한 현장 상황 입력 → Draft 저장 및 Review 제출 → 요약 리포트 산출"의 End-to-End 동작 시연 완료.

## 12. 후속 시스템 영향 및 연결성

1. **DATA-SCHEMA**: 상태 전이를 감당할 `AuditSession`, 템플릿 버전을 추적할 `AuditTemplate`, 항목별 결과를 담을 `AuditResultItem` 테이블 정규화 필요.
2. **COM-AUTH**: Reviewer 권한의 승인 행위에 대한 RBAC 분리.
3. **DATA-AUDIT_LOG**: 모든 상태 전이 트랜잭션 성공 직후 `AuditLog` Insert 트리거 연동.
4. **ADM-BULK**: 본 모듈에서 사용할 Template, Site, 장비 마스터 데이터가 Bulk Import를 통해 선행 적재되어야 함.

## 16. Opus 4.6 리뷰 포인트

다음 9가지 점검 항목을 기반으로 본 문서의 설계 안정성을 검토합니다.

1. **감사 상태 전이의 누락 여부**: Draft ➔ Review ➔ Approved ➔ Report Generated의 1-Way 흐름 중 교착 상태(Deadlock)에 빠질 수 있는 예외(예: 결재자 부재) 고려가 되었는가?
2. **역할별 권한 경계의 불명확성**: 심사원(Auditor)이 스스로 승인(Approver 권한 획득)하는 셀프 결재를 기술적으로 명확히 분리/차단하였는가?
3. **STT 의존도가 과도한지 여부**: "STT는 보조 수단"이라는 대원칙에 따라, 음성 인식 서버가 완전히 다운된 최악의 상황에서도 터치/타이핑으로 감사가 100% 완수될 수 있도록 설계(Fallback)되었는가?
4. **Template 구조가 너무 약하거나 복잡한가**: Sprint 1 범위에서 불필요한 복합 룰엔진(분기/로직)을 덜어내고, Section-Item 단위의 직관적인 동적 폼 렌더링에 집중했는가?
5. **리포트 생성 흐름의 현실성**: 승인 즉시 동기 방식으로 생성할지, 비동기 큐를 탈지 트랜잭션 처리 전략이 문서에 명확한가 (Sprint 1은 동기 또는 즉각 비동기로 10분 내 시연 보장).
6. **감사 로그 이벤트 누락 여부**: 단순 '저장'뿐만 아니라 '승인', '반려', '접근 거부' 등 규제 방어에 필수적인 보안 이벤트가 `DATA-AUDIT_LOG`로 넘어가는 트리거 포인트가 확실한가?
7. **Sprint 1 범위 과대 여부**: 결함 사진 분석(Vision AI)이나 NC 연계를 "할 수 있으면 좋음"이라는 이유로 Sprint 1에 끼워 넣어 일정을 위협하지 않았는가?
8. **API / UI / TEST 연결성 부족 여부**: 상태 값, 권한 체계 표가 백엔드 enum이나 프론트 라우팅 가드로 1:1 매핑될 수 있을 만큼 명확한 영문 키워드와 조건으로 작성되었는가?
9. **현장 감사자가 쓰기 어려운 설계인가**: 다단계의 팝업, 과도한 필수 값, 잦은 화면 전환 등 모바일/태블릿 환경에서 작업 피로도를 높이는 UI/UX 제약이 강제되지 않았는가?
