# 작업 목록 (TASK LIST) v2.0 - 운영 최적화 버전

## 1. 문서 목적
본 문서는 PRO ILI SMART 프로젝트의 기존 TASK LIST(v1.0)를 바탕으로, 개발, QA, 운영팀이 즉각적이고 투명하게 시스템을 검증할 수 있도록 **"실제 운영 가능한 수준(Implementation-ready)"**으로 재구성한 버전이다. 모든 Task는 정량적 완료 기준을 가지며, AI 품질 검증, 실패 대응, Observability(관측 가능성) 및 규제 대응을 완벽히 포괄한다.

---

## 1) 기존 Task 평가 요약

기존 TASK LIST v1.0의 작업 항목들에 대한 품질 평가 결과이다. 모든 기존 태스크의 완료 기준이 "테스트 통과", "동작 완료" 등 추상적인 언어로 작성되어 있어 전면적인 **IMPROVE(보강)** 처리를 진행했다. 또한 운영 안정성과 품질 보증을 위해 5개의 신규 Task 카테고리를 **MISSING(신규 추가)**으로 식별하여 보강하였다.

*   **[IMPROVE] T1-001 ~ T4-005 (전체 25개 기존 Task)**
    *   **이유**: 정량적 지표 부재, 실패 조건 미정의, 검증 방법 불명확.
    *   **조치**: 각 Task마다 정량 지표, 실패 조건, 검증 방법을 명시하여 강제 보강. 특히 AI Task(T1-006, T1-008 등)에 정확도 지표(WER, 매핑율)와 실패 케이스 분류 체계 의무화.
*   **[MISSING] AI 품질 검증 및 Golden Dataset (신규 T1-010, T1-011)**
    *   **이유**: AI 정확도 측정을 위한 정답지(Dataset)와 이를 평가할 파이프라인 누락.
    *   **조치**: Golden Dataset 구축 및 자동 검증 파이프라인 Task 신설.
*   **[MISSING] Observability / KPI 측정 (신규 T1-012)**
    *   **이유**: Audit 시간 단축, STT Fallback 등 북극성 KPI를 추적하고 에러를 캡처할 스키마 부재.
    *   **조치**: 로그 스키마 정의 및 에러 추적 시스템 구축 Task 신설.
*   **[MISSING] 실패 대응 / Fallback UX (신규 T1-013)**
    *   **이유**: AI 인식 실패 시 사용자의 작업 이탈을 막을 시나리오 부재.
    *   **조치**: 강제 수동 입력 전환 등 Fallback UX 로직 추가.
*   **[MISSING] 규제 및 데이터 수집 동의 (신규 T1-014)**
    *   **이유**: PIPA 동의 로직은 Sprint 4에 있으나, DB 설계 및 동의 로그 아키텍처는 Sprint 1부터 적용되어야 함.
    *   **조치**: 데이터 수집 동의 이력 로그 DB 처리 Task를 기반 공사에 추가.

## 3) 신규 Task만 별도 요약 (MISSING 반영분)

빠른 식별과 리소스 할당을 위해 새롭게 추가된 5개의 코어 Task를 분리 요약한다. 해당 Task들은 시스템의 안정성, AI 성능의 객관적 검증, 그리고 법적 리스크 방어의 핵심이 된다.

1.  **T1-010 (Golden Dataset 구축)**
    *   **목적**: "AI가 잘 된다"는 주관적 평가를 배제하고, 모델 성능을 수치화할 정답지 세트를 구축한다.
    *   **완료 기준**: STT 100건, 매핑 50건의 정답지를 확보하고 DVC에 버전 등록 100% 완료.
2.  **T1-011 (AI 품질 검증 파이프라인)**
    *   **목적**: 향후 프롬프트나 모델 교체 시 즉각적인 회귀(Regression)를 잡기 위해 자동 채점 로직을 도입한다.
    *   **완료 기준**: 정답지 대비 F1-Score 산출 스크립트를 GitHub Actions에 연동하여 CI/CD 방어망 구축.
3.  **T1-012 (Observability 로그 스키마)**
    *   **목적**: Audit 소요 시간, STT fallback율 등 비즈니스 KPI를 측정하기 위한 계측 로직을 선반영한다.
    *   **완료 기준**: API 에러 및 AI Fallback 발생 후 1초 이내 로그 적재, 누락률 0%.
4.  **T1-013 (실패 대응 Fallback UX)**
    *   **목적**: 소음/방언 등으로 STT가 완전히 실패했을 때, 사용자가 화를 내고 앱을 끄지 않도록 수동 전환 루트를 만든다.
    *   **완료 기준**: STT 실패 판정 시 500ms 내 수동 입력 UI 전환, 이탈율 방어.
5.  **T1-014 (데이터 수집 동의 이력 로그)**
    *   **목적**: 민감 정보 수집에 대한 법적 분쟁을 방어하기 위해 동의 여부를 변경 불가능한 DB에 영구 기록한다.
    *   **완료 기준**: 가입 시 PIPA 동의 내역을 DB에 적재, 동의 거부 시 앱 사용 원천 차단 로직(E2E) 완료.
