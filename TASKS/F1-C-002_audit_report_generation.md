# [F1-C-002] Audit Report Generation 처리 명세서

## 1. 문서 목적
이 문서는 PRO ILI SMART 프로젝트의 F1 Smart Audit 도메인에서, 감사가 완료된 세션(Audit Session) 데이터를 기반으로 **공식 감사 리포트(Audit Report)**를 생성하고 관리하는 제반 처리(Command) 로직을 표준화하기 위해 작성되었다.

- **문서 필요성**: 세션이 '현장의 땀방울(작업)'이라면 리포트는 '최종 결론(얼굴)'이다. 어떤 조건에서 리포트가 만들어지고 갱신되는지 엄격히 정의해야 한다.
- **Sprint 1 중요성**: 핵심 KPI인 "10분 내 리포트 산출"을 달성하기 위한 최소 기능 요건을 명확히 하고 오버스펙을 차단한다.
- **후속 문서와의 관계**: 본 문서는 리포트가 '만들어지는 과정'에 집중하며, 만들어진 리포트를 조회하는 `F1-Q-001_audit_report_query.md`와 API 처리를 담당할 `F1-RH-001`, `API-003` 등의 상위 기준(Ground Truth)이 된다.

## 4. Audit Report 개념 정의

- **한 줄 정의**: 완료된 `Audit Session`의 스냅샷 데이터를 바탕으로, 감사 대상, 점수, 부적합 내역, 증빙 자료 등을 일목요연하게 정리한 **불변(Immutable) 공식 산출물**.

## 5. 리포트 생성 전제조건

**원칙**: 리포트가 생성되려면 세션의 상태와 무결성이 완벽하게 보장되어야 한다.

### 표 4. 리포트 생성 전제조건
| 항목 | 설명 | 필수 여부 | 검증 포인트 | 비고 |
| :--- | :--- | :--- | :--- | :--- |
| **세션 상태 검증** | 세션이 반드시 `Finalized` 상태여야 함 (Sprint 1 정책에 따라 `Submitted` 허용 가능성 확인 필요) | Y | DB 내 `Session.state == 'Finalized'` | 미완성본 차단 |
| **필수 데이터 완료** | 템플릿에 정의된 모든 Mandatory 항목에 값이 있어야 함 | Y | Payload Validation | `Finalized` 전이 시점에 이미 검증됨 |
| **Tenant/Site 일치** | 리포트 생성을 요청한 주체와 세션의 테넌트/사이트가 일치해야 함 | Y | JWT Claim vs Session DB Record | 데이터 유출 원천 차단 |
| **메타데이터 존재** | 감사 대상, 일자, 감사자 ID 등 필수 메타데이터가 있어야 함 | Y | Null Check | 문서의 헤더 구성 필수값 |
| **중복 생성 방지** | 이미 해당 세션의 동일 버전 리포트가 `Generated` 상태라면 재생성 안 함 | Y | `Report.session_id` && `Report.version` 체크 | 멱등성 보장 |

## 6. 리포트 생성 흐름

### 표 5. 리포트 생성 흐름
| 단계 | 설명 | 입력 | 처리 | 출력 | 실패 시 대응 | Audit Log |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **1. 요청 수신** | 리포트 생성 API 호출 (자동/수동) | `session_id` | 테넌트/권한/세션상태 검증 | 검증 통과 여부 | 400/403/404/409 | Y (실패 시에도) |
| **2. 데이터 수집** | 세션의 최종 Payload, Template 구조, 첨부파일 URL 등 수집 | `session_id`, `template_id` | DB Join 및 JSON Fetch | Raw Session Payload | 500 내부 에러 (DB 연결 등) | N |
| **3. 데이터 변환/결합** | Template 구조에 맞게 세션 데이터를 배치하고 이슈 추출 | Raw Data | 그룹핑, 부적합 필터링, 요약 텍스트 추출 | Report JSON Object | 500 데이터 정합성 에러 | N |
| **4. 산출물 생성** | JSON 데이터를 Report Table에 Insert, (옵션) PDF 생성 큐 전송 | Report JSON | DB Insert (상태: `Generating` -> `Generated`) | `report_id` | 상태 `Failed` 로 업데이트 | Y |
| **5. 구버전 처리** | 재성성인 경우 기존 리포트를 `Superseded`로 무효화 | `report_id` | 기존 Report 레코드 상태 업데이트 | - | 트랜잭션 롤백 | N (4번과 트랜잭션 묶임) |

## 7. 리포트 상태 정의

### 표 6. 리포트 상태 정의
| 상태 (State) | 의미 | 진입 조건 | 종료 조건 | 허용 액션 | 비고 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **DraftReport** | (사용 안 함) 세션이 Draft이므로 리포트는 이 상태를 갖지 않음 | - | - | - | 논리적으로 배제 |
| **Generating** | 리포트 JSON 변환 및 PDF 생성이 진행 중인 상태 | 생성 전제조건 통과 직후 | JSON 및 파일 저장 완료 | 조회 대기 | 비동기 생성 시 중요 |
| **Generated** | 공식 리포트 산출물이 성공적으로 생성되어 열람 가능한 상태 | `Generating` 정상 완료 | 세션이 Reopen되어 재생성될 때까지 | 조회, 다운로드 | **정상(최종) 상태** |
| **Failed** | 생성 과정에서 시스템 에러로 실패한 상태 | `Generating` 중 Exception 발생 | 관리자의 재시도(Retry) 시 | 생성 재시도 | 운영 모니터링 대상 |
| **Regenerating** | 기존 리포트를 무효화하고 새로 만들고 있는 상태 | 수동 재생성 요청 시 | 완료 시 `Generated` | 조회 대기 | |
| **Superseded** | 세션 수정 등으로 새 리포트가 나와 효력을 잃은 구버전 상태 | 신규 버전이 `Generated` 될 때 | 없음 (영구 보관) | 조회 (이력 확인용) | 구버전 다운로드 제한 권장 |
| **Archived** | 오래된 리포트가 콜드 스토리지로 이동된 상태 | 정책에 따른 스케줄러 실행 | 복원 요청 시 | 제한적 조회 | Sprint 1에서는 미고려 |

## 10. 재생성 및 버전 관리 기준

**원칙**: 리포트는 한 번 확정되면 불변(Immutable)이지만, 감사 현장의 사후 조치 등으로 세션 수정이 불가피한 경우 "기존 문서를 무효화하고 새 버전을 발행"하는 구조를 따른다.

### 표 9. 재생성 및 버전 관리 기준
| 상황 | 설명 | 허용 여부 | 처리 방식 | 비고 |
| :--- | :--- | :--- | :--- | :--- |
| **세션 수정(Reopen) 후 제출** | `Finalized` 세션이 Reopen되어 내용 수정 후 다시 `Finalized`됨 | Y | 기존 리포트를 `Superseded`로 변경 후 버전 올려(ex: v2) 새 리포트 `Generated` | 이력 추적 유지 |
| **단순 재생성 요청** | 세션 데이터 변경 없이 포맷 오류 등으로 관리자가 재생성 트리거 | Y | 기존 리포트 덮어쓰기 또는 버전업 (정책에 따름) | 예외적 상황 대응용 |
| **최신본 기준 (Latest)** | 다수의 버전이 존재할 때 API 조회 기준 | - | 항상 가장 높은 버전의 `Generated` 상태 리포트만 `is_latest=true` 취급 | 혼선 방지 |

| **Other Employee** | N | N | 제한적 (읽기권한자) | 제한적 | 권한 부여된 건만 |
| **Guest / None** | N | N | N | N | 접근 불가 (403) |

> [!NOTE]
> Sprint 1에서는 Auditor가 세션을 `Submit` (혹은 `Finalize`) 하는 순간 **시스템이 자동으로 리포트 생성을 트리거**하는 Event-driven 방식이 주를 이루며, 수동 생성 API는 주로 Admin용 복구 도구로 활용된다.

## 12. 오류 및 예외 처리

### 표 11. 오류 및 예외 처리 기준
| 오류 유형 | 설명 | 발생 시점 | 사용자 대응 (UI) | 운영자 대응 | 공통 에러 스키마 (API-001) |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **전제조건 미달** | 세션이 Finalized가 아닌데 생성 요청 | API 호출 시 | "리포트를 생성할 수 없는 상태입니다." | - | 400 (`INVALID_STATE_TRANSITION`) |
| **권한 오류** | 권한 밖의 리포트 생성/조회 요청 | API 호출 시 | "접근 권한이 없습니다." | 비정상 접근 패턴 로깅 | 403 (`ACCESS_DENIED`) |
| **데이터 결함** | 세션 Payload가 깨졌거나 템플릿과 불일치 | 변환(Generator) 중 | "리포트 생성 중 데이터 오류가 발생했습니다." | Raw Payload 및 DB 무결성 확인 | 500 (`DATA_CORRUPTION`) |
| **생성 타임아웃** | PDF 렌더링 서버 등 지연 발생 | 생성 과정 중 | "리포트 생성이 지연되고 있습니다." (Polling/Websocket 안내) | 인프라/큐 모니터링 | 504 (`GATEWAY_TIMEOUT`) |

## 13. 감사 로그 및 추적 연계

**원칙**: 리포트라는 공식 산출물이 만들어지고, 무효화되는 모든 이벤트는 엄격하게 기록된다.

### 표 12. 감사 로그 및 추적 연계 기준
| 이벤트 (Action) | 설명 | 기록 필드 (Payload) | 중요도 | 관련 문서 |
| :--- | :--- | :--- | :--- | :--- |
| `REPORT_GENERATION_STARTED`| 리포트 생성 프로세스 개시 | `session_id`, `actor_id` | Medium | - |
| `REPORT_GENERATED` | 리포트 산출물 완성 | `report_id`, `version`, `session_id` | High | 리포트 수명주기의 시작점 |
| `REPORT_GENERATION_FAILED`| 리포트 생성 중 시스템 에러 | `session_id`, `error_cause` | Critical | 얼럿 연계 (NFR-MON-002) |
| `REPORT_SUPERSEDED` | 구버전 리포트 무효화 | `report_id`, `new_report_id` | High | 문서 무결성 추적용 |
| `REPORT_DOWNLOADED` | 사용자가 PDF 등을 다운로드 | `report_id`, `actor_id`, `ip_address` | High | 문서 유출 보안 추적용 |

## 16. 완료 기준 (Definition of Done)

본 문서의 작성 및 이관이 완료되었다고 판단하는 기준은 다음과 같다.

1. **책임 분리 명확화**: 리포트 생성이 세션 입력 과정과 철저히 분리된 독립된 후속 프로세스임이 명시됨.
2. **전제조건 확립**: 어떤 세션(상태, 권한 등)만이 리포트로 승격될 수 있는지 차단막(표 4, 11)이 명확히 정의됨.
3. **버전 관리 규칙화**: 현장의 데이터 수정 요구를 반영하면서도 산출물 무결성을 지키는 재생성/Superseded 규칙이 정의됨.
4. **구성 요소 도출**: 클라이언트가 리포트를 화면에 그리기 위해 백엔드에 기대하는 JSON 구조(섹션들)가 표 8에 도출됨.
5. **승인 대기(Ready for Next)**: 백엔드가 쿼리와 API 스펙(`F1-Q-001`, `API-003`)을 설계할 수 있도록 명세 제공 완료.
