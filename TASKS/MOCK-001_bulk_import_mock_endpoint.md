# [MOCK-001] Bulk Import Mock Endpoint 기준 명세

## 1. 문서 목적

*   **문서 필요성:** 프론트엔드와 QA 파트가 실제 백엔드(Route Handler 및 비즈니스 로직) 구현 완료를 기다리지 않고, 사전에 정의된 규격대로 병렬 개발 및 테스트 시나리오 검증을 진행할 수 있도록 Bulk Import API의 가짜(Mock) 응답 기준을 정의한다.
*   **Sprint 1 중요성:** Bulk Import는 Sprint 1의 핵심 기능이다. 병목 현상(Bottleneck)을 방지하기 위해 성공/부분 성공/실패 등 다양한 상태 전이를 시뮬레이션할 수 있는 환경이 조기에 확보되어야 한다.
*   **실제 서버 구현 전 필요성:** 단순한 "200 OK" 응답이 아닌, UI 상태(배지, 프로그래스바, 드릴다운)와 QA의 예외 시나리오(권한 차단, 재처리)를 재현할 수 있는 정교한 Mock이 있어야 프론트엔드 개발의 완성도를 높일 수 있다.
*   **후속 구현과의 관계:** 이 문서에서 정의된 응답 구조와 에러 스키마는 곧 실제 `COM-RH-001`(Route Handler)이 반환해야 할 실제 데이터의 청사진이 된다.

## 12. UI / TEST / Route Handler 영향

*   **`UI-061` (화면 개발):** 이 Mock API 규격이 준비되면, 프론트엔드 개발자는 백엔드 서버 구동 없이 `npm run dev` 만으로 100% 화면 레이아웃과 상태 전환을 구현할 수 있다.
*   **`TEST-ADM-001` (QA 테스트):** QA 파트는 Mock 서버를 타겟으로 Playwright 등 E2E 테스트 스크립트를 작성하여 자동화 테스트를 미리 구축할 수 있다.
*   **`COM-RH-001` (백엔드 구현):** 백엔드 개발자는 본 Mock의 JSON 반환 형태를 보며, 실제 Route Handler에서 Response DTO를 어떻게 조립해야 할지 구체적인 역엔지니어링(Reverse Engineering) 힌트를 얻는다.
*   **실제 전환 시:** API Endpoint의 도메인(Base URL)만 변경하면 Mock에서 실제 서버로 100% 호환 전환되어야 한다.

## 14. 완료 기준 (Definition of Done)

*   **문서 완료 조건:** 표 1~12의 내용이 누락 없이 작성되고, 성공/부분 성공/실패/상태전이 Mock 전략이 구체적으로 정의됨.
*   **Mock 범위 정의 완료 조건:** 프론트엔드 개발자가 이 문서를 읽고 MSW(Mock Service Worker) 또는 API Mock 서버의 Handler 코드를 즉시 작성할 수 있음.
*   **UI/QA 전달 가능 조건:** 문서의 Mock 시나리오만으로 UI 디자인의 모든 Edge Case(빈 상태, 에러 상태, 로딩 상태)를 재현할 수 있음.
*   **실제 전환 가능 조건:** Mock 응답 구조가 `API-002` 및 `API-001`과 100% 일치하여, 베이스 URL 변경만으로 실제 서버 연동에 문제가 없음을 보장함.
